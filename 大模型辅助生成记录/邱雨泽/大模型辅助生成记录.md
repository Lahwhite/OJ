# 题目管理模块大模型辅助生成记录

## 一、基本信息

- 记录人：邱雨泽
- 负责模块：题目管理模块（Problems）
- 辅助工具：大语言模型辅助编程工具
- 辅助范围：需求分析、前后端实现、数据库设计、联调测试、文档整理、质量检查与报告生成

## 二、辅助生成内容概述

在题目管理模块开发过程中，我多次使用大模型辅助完成代码阅读、方案设计、代码生成、问题定位和文档整理。大模型主要承担“分析现有项目结构、给出实现建议、生成初版代码、辅助调试和总结材料”的角色，最终代码仍结合项目实际运行结果进行检查和修改。

## 三、主要辅助过程

### 1. 仓库拉取与项目结构理解

使用大模型辅助拉取和更新 GitHub 仓库，阅读项目中其他模块的前端代码风格，包括用户、排行榜、讨论区等模块的页面布局、接口调用方式和静态资源组织方式。在此基础上，确定题目管理模块前端应保持统一的视觉风格和交互方式。

### 2. 题目管理前端实现

根据已有模块风格，辅助完成题目管理模块的前端页面设计与实现，包括题目列表、题目详情、创建题目、编辑题目、删除题目、分页展示、难度标签和 AC 状态展示等功能。后续根据实际页面效果，修复了题目列表中文本堆叠、卡片布局异常等问题，使页面展示更加接近其他模块。

### 3. 后端接口与数据库设计

借助大模型梳理题目管理模块后端接口，设计并完善题目、测试用例、用户题目状态等相关数据表。数据库部分主要完成了题目信息存储、测试样例存储、用户 AC 状态、历史最高分、最近提交时间等字段设计。同时，为避免与其他模块的用户表冲突，将本模块用户表调整为 `problem_users`，并精简不需要的邮箱、头像、密码哈希等字段。

### 4. AC 状态存储与展示

围绕“题目是否完成 AC”的需求，大模型辅助分析了由哪个模块维护测评结果更合适。最终确定题目管理模块只负责 AC 信息的存储和展示，更新操作由其他模块或评测结果触发。实现后，题目列表中可以类似洛谷题单样式显示通过状态：通过显示绿色对勾，未通过显示横杠或未通过状态。

### 5. 与评测模块联动

大模型辅助阅读评测模块代码，确认评测模块会在测评结束后输出结果 JSON 路径，例如 `Result json saved to:` 和 `OJ_RESULT_JSON=`。在此基础上，为题目管理模块补充了读取评测结果 JSON 的逻辑：当本模块发起代码提交并收到评测结果路径后，解析 JSON 中的 `verdict`、`total_score`、`max_score` 等字段，并更新数据库中的 AC 状态、最高分和最近提交时间。

### 6. 本地运行与环境配置

开发过程中多次使用大模型辅助启动项目，检查 Maven、Spring Boot、MySQL、H2 测试环境和配置文件。针对本地 MySQL 用户名和密码配置、默认配置恢复、端口访问路径 `/api/` 修改为 `/problems/` 等问题进行了排查和说明。同时定位过 Maven clean 无法删除日志文件、进程占用、启动卡住、Windows 弹窗错误等问题。

### 7. 测试与质量保障

使用大模型辅助执行和分析 Maven 测试、JaCoCo 覆盖率、PMD 静态分析等质量检查。为新增的评测结果同步逻辑补充了单元测试，验证题目管理模块能够识别评测输出路径、读取 JSON，并调用状态更新服务。当前 Problems 模块测试结果为 45 个测试全部通过。

### 8. 文档与报告生成

大模型辅助整理了题目管理模块 README、接口说明、数据库说明、个人报告、UML 类图、代码行数统计、注释率说明、缺陷修复统计等材料。文档内容经过多次精简和调整，使其更适合课程提交和项目展示。

## 四、典型问题与辅助修复

| 问题类型 | 问题描述 | 处理方式 |
| --- | --- | --- |
| 前端布局问题 | 题目列表文字堆叠、卡片内容挤压 | 调整 CSS 布局、间距和列表结构 |
| 数据库冲突 | `users` 表可能与其他模块重复 | 修改为 `problem_users` 并精简字段 |
| AC 状态缺失 | 初始版本只能管理题目，不能展示用户完成状态 | 新增题目状态表和状态查询接口 |
| 模块联动问题 | 评测结果由 judge 产生，Problem 模块无法直接判断 AC | 读取 judge 输出 JSON 并更新本模块状态 |
| 启动失败 | MySQL 配置、日志文件占用、进程残留导致启动异常 | 临时修改配置验证，清理占用进程后恢复默认配置 |
| 测试覆盖 | 新增功能需要验证 | 添加针对评测结果同步的单元测试 |

## 六、个人总结

大模型在本模块开发中主要起到了快速理解项目、生成初版实现、辅助排错和整理文档的作用。它提高了代码编写和问题定位效率，尤其在跨模块联动、数据库设计和测试补充方面帮助较大。但在实际使用中仍需要人工判断方案是否符合项目边界，例如不能随意修改其他模块、不能让题目管理模块承担代码测评职责等。最终实现过程中，我需要结合运行结果、测试输出和项目规范，对大模型生成内容进行筛选、修改和验证。
